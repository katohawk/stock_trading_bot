# 加密货币自动交易：高抛低吸与快速操作

在现有「数据 → 策略 → 风控 → 回测/实盘」架构下，补充**加密货币（如比特币）**的自动交易思路，重点实现**高抛低吸**和**快速操作**。

---

## 一、与股票的区别

| 维度       | 股票                     | 加密货币（如 BTC）                    |
|------------|--------------------------|--------------------------------------|
| 交易时间   | 交易日盘前/盘中/盘后     | **24/7 连续**                        |
| 波动与流动性 | 涨跌停、T+1 等           | **波动大、流动性好**，适合短线频繁交易 |
| 数据与接口 | 券商/第三方行情与下单   | **交易所 REST + WebSocket** 统一     |
| 手续费     | 佣金+印花税等            | **Maker/Taker 费率**，部分免佣      |
| 标的       | 多标的、需选股           | 单标的或少标的即可（如 BTC/USDT）   |

因此：加密货币更适合**高抛低吸、快速多次操作**，需要**实时行情 + 低延迟下单**。

---

## 二、“高抛低吸”怎么做

**高抛低吸** = 在相对低位买、相对高位卖，反复做价差，不依赖长期单边趋势。

### 1. 网格策略（最直接）

- 设定一个**价格区间** [low, high]，在区间内等分若干档（或按百分比分档）。
- **跌到某档就买**，**涨到某档就卖**，每档对应固定数量或固定金额。
- 适合**震荡市**；单边大涨会卖飞，单边大跌会抄底抄在半山腰，需要设总仓位/止损上限。

### 2. 均值回归

- 用**均线或布林带**判断价格偏离程度，偏离下轨买、偏离上轨卖。
- 可与 K 线周期结合：周期越短操作越频繁，但对延迟和手续费越敏感。

### 3. 区间 + 条件单

- 先根据历史波动率或 ATR 估计**近期波动区间**，在区间内设置多档挂单（限价单）。
- 交易所支持**条件单/计划委托**时，可预先挂好“跌破 X 买”“涨破 Y 卖”，实现快速反应。

### 4. 快速操作的实现要点

- **行情**：用交易所 **WebSocket** 拿实时 tick / K 线，避免 REST 轮询延迟。
- **下单**：REST 或 WebSocket 下单接口，选**延迟低的区域/节点**；大单可拆成多笔减小冲击。
- **周期**：高抛低吸常用 **1m / 5m / 15m** K 线；周期越短，信号越多，但对延迟和手续费越敏感。
- **幂等与风控**：同一逻辑只下一笔单、带 order_id 防重；总仓位、单日亏损、单标的敞口要有上限。

## 三、在现有架构中的位置

```
                    ┌─────────────────────────────────────────┐
                    │  加密货币：交易所 API（REST + WebSocket） │
                    └─────────────────────────────────────────┘
                                          │
  ┌─────────────┐    ┌─────────────┐    ▼    ┌─────────────┐
  │ 数据层      │    │ 策略层      │         │ 风控层      │
  │ CryptoAdapter│──▶│ 网格/均值回归│────────▶│ 仓位/止损   │
  │ (K线/实时)   │    │ 只出信号    │         │ 熔断        │
  └─────────────┘    └─────────────┘         └─────────────┘
                                                      │
  ┌─────────────┐    ┌─────────────┐                 ▼
  │ 回测        │    │ 执行层      │         ┌─────────────┐
  │ 历史K线     │    │ 模拟/实盘   │◀────────│ 目标仓位    │
  └─────────────┘    │ 交易所API   │         │ 或订单列表  │
                     └─────────────┘         └─────────────┘
```

- **数据层**：新增加密货币数据源（如 ccxt / 交易所直连），输出与现有 **Bar** 一致的 K 线；实盘可再接 WebSocket 实时价。
- **策略层**：新增**网格策略**（及可选均值回归），只输出**信号**（买/卖、数量或比例），不直接下单。
- **风控层**：复用现有仓位、止损、熔断；可按标的或“总敞口”限制加密货币仓位。
- **执行层**：回测用模拟成交；实盘对接交易所 API（REST/WebSocket），同一套“信号 → 目标仓位/订单”逻辑。

---

## 四、推荐实现顺序

1. **数据**：实现 **CryptoAdapter**，拉取 BTC/USDT 等历史 K 线（1m/5m/1h），统一成 Bar；实盘再补 WebSocket 实时价。
2. **策略**：实现**网格策略**（区间、档位、每档数量），只输出信号；可选加简单均值回归（如布林带）。
3. **回测**：用历史 K 线 + 网格策略 + 现有风控与手续费/滑点，得到收益曲线与最大回撤、胜率等；重点看**震荡段**表现。
4. **实盘**：用**模拟盘**或小资金，同一套“策略+风控”接交易所 API；先保证信号→下单稳定、无重复下单，再考虑延迟与拆单。

---

## 五、风险与注意

- **24/7 波动**：极端行情下网格可能爆仓或巨亏，必须设总仓位上限、止损和熔断。
- **交易所与 API**：选主流交易所（如 Binance、OKX），注意 API 限频、IP 限制和合规。
- **手续费**：高频时手续费会吃掉利润，回测和实盘都要用真实 Maker/Taker 费率。
- **不保证盈利**：高抛低吸在单边行情中容易失效，需充分回测和模拟后再上实盘。

## 六、本仓库中的用法

- **数据**：`src.data.CryptoAdapter`（ccxt）拉取交易所 K 线；若未装 ccxt 或接口异常，`run_crypto_backtest.py` 会退化为 yfinance 的 BTC-USD 日线做演示。
- **策略**：`src.strategy.GridStrategy`（区间 + 档位），回测引擎已支持按目标仓位加减仓。
- **运行**：`python run_crypto_backtest.py`；实盘需自行对接交易所 REST/WebSocket 下单并替换执行层。
